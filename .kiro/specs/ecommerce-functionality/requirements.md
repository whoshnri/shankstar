# Requirements Document

## Introduction

This document defines the requirements for the E-Commerce Functionality feature of the Next.js luxury store application. The feature covers six interconnected areas: Prisma v7 upgrade, a Server Actions data layer, Customer schema with checkout integration, a Search overlay with URL routing, a responsive Cart sheet fix, and a cookie-based Wishlist. Together these replace all mock data with real database queries and deliver the remaining UI features needed for a complete e-commerce experience.

## Glossary

- **Server_Action**: An async TypeScript function marked with `"use server"` that runs exclusively on the server and is callable from React Server Components or client components via `useTransition`.
- **PrismaClient**: The database client generated by Prisma ORM used to query PostgreSQL.
- **Singleton**: A design pattern ensuring at most one instance of a class exists per process.
- **Customer**: A person who places an order; identified uniquely by email or phone number.
- **Order**: A confirmed purchase record linking a Customer to one or more OrderItems.
- **OrderItem**: A single line in an Order representing a quantity of a specific ProductVariant.
- **ProductVariant**: A specific SKU of a Product (e.g., colour + size combination) with its own price and stock count.
- **Wishlist**: A device-local list of ProductVariants saved in a browser cookie.
- **WishlistItem**: A single entry in the Wishlist containing variant, product, name, image, price, and slug.
- **SearchOverlay**: The client-side search input component rendered in the site header.
- **Search_Page**: The React Server Component at `/search/[query]` that displays search results.
- **CartSheet**: The slide-in panel that displays the current shopping cart contents.
- **Drawer**: The vaul bottom-slide component used for the cart on mobile viewports.
- **Sheet**: The shadcn/ui right-side slide-in panel used for the cart on desktop viewports.
- **Zod**: A TypeScript-first schema validation library used to validate Server Action inputs.
- **ILIKE**: A case-insensitive SQL LIKE operator available in PostgreSQL.
- **upsertCustomer**: The Server Action that creates or updates a Customer record based on email or phone match.
- **createOrder**: The Server Action that atomically creates an Order and its OrderItems while decrementing stock.
- **Coupon**: A discount code with optional expiry, usage limit, and discount type (percentage or fixed).

## Requirements

### Requirement 1: Prisma v7 Configuration

**User Story:** As a developer, I want the Prisma client to be generated to a local output path and imported via a singleton, so that the application works correctly with Prisma v7 and avoids multiple client instances.

#### Acceptance Criteria

1. THE `schema.prisma` generator block SHALL set `provider` to `"prisma-client-js"` and `output` to `"../prisma/generated/prisma"`.
2. THE `lib/prisma.ts` module SHALL import `PrismaClient` from `"../prisma/generated/prisma"`.
3. THE `lib/prisma.ts` module SHALL use `globalThis` to cache the `PrismaClient` instance so that at most one instance exists per Node.js process.
4. WHEN `NODE_ENV` is not `"production"`, THE `lib/prisma.ts` module SHALL assign the `PrismaClient` instance to `globalThis` to survive hot-module reloads.
5. WHEN `NODE_ENV` is `"production"`, THE `lib/prisma.ts` module SHALL create exactly one `PrismaClient` instance without assigning it to `globalThis`.

### Requirement 2: Server Actions — Products

**User Story:** As a developer, I want typed Server Actions for product data retrieval, so that pages and components can fetch real product data without client-side waterfalls.

#### Acceptance Criteria

1. THE `lib/actions/products.ts` module SHALL export a `getProducts` function that accepts an optional `categorySlug` and `limit` and returns a `Product[]`.
2. WHEN `getProducts` is called with a `categorySlug`, THE `products` action SHALL return only products whose category slug matches the provided value.
3. THE `lib/actions/products.ts` module SHALL export a `getProductBySlug` function that returns a `ProductWithVariants` or `null`.
4. THE `lib/actions/products.ts` module SHALL export a `getFeaturedProducts` function that accepts an optional `limit` and returns a `Product[]`.
5. THE `lib/actions/products.ts` module SHALL export a `getRecommendations` function that accepts a `productId` and optional `limit` and returns a `Product[]` excluding the source product.
6. WHEN any products action is called, THE `products` action SHALL only return products where `isVisible` is `true`.

---

### Requirement 3: Server Actions — Search

**User Story:** As a developer, I want a `searchProducts` Server Action, so that the search page can query products by name, brand, and category with recommendations.

#### Acceptance Criteria

1. THE `lib/actions/search.ts` module SHALL export a `searchProducts` function that accepts a `query` string and returns `{ results: Product[], recommendations: Product[] }`.
2. WHEN `searchProducts` is called with a query of fewer than 2 characters, THE `search` action SHALL return `{ results: [], recommendations: [] }`.
3. WHEN `searchProducts` is called with a valid query, THE `search` action SHALL match products whose `name`, `brand.name`, or `category.name` contains the query string (case-insensitive).
4. WHEN `searchProducts` is called with a valid query, THE `search` action SHALL return only products where `isVisible` is `true`.
5. FOR ALL valid queries, THE `search` action SHALL return `recommendations` that contain no product already present in `results`.
6. WHEN `searchProducts` returns results, THE `search` action SHALL populate `recommendations` with up to 8 visible products from the same categories as the results, ordered randomly.
7. WHEN `searchProducts` returns no results, THE `search` action SHALL still return a non-empty `recommendations` array drawn from all visible products.

---

### Requirement 4: Server Actions — Customers

**User Story:** As a developer, I want an `upsertCustomer` Server Action, so that checkout can create or retrieve a Customer record without creating duplicates.

#### Acceptance Criteria

1. THE `lib/actions/customers.ts` module SHALL export an `upsertCustomer` function that accepts a `CustomerInput` object and returns a `Customer`.
2. WHEN `upsertCustomer` is called with an email and phone that do not match any existing Customer, THE `customers` action SHALL create and return a new Customer record.
3. WHEN `upsertCustomer` is called with an email that matches an existing Customer, THE `customers` action SHALL update and return that existing Customer record.
4. WHEN `upsertCustomer` is called with a phone that matches an existing Customer, THE `customers` action SHALL update and return that existing Customer record.
5. WHEN `upsertCustomer` is called, THE `customers` action SHALL normalize the email to lowercase and trimmed form before storing.
6. WHEN `upsertCustomer` is called, THE `customers` action SHALL normalize the phone to digits-only (E.164-style) before storing.
7. FOR ALL `upsertCustomer` calls sharing the same normalized email, THE `customers` action SHALL return a Customer with the same `id`.
8. WHEN `upsertCustomer` receives invalid input, THE `customers` action SHALL reject the call with a Zod validation error before performing any database operation.

### Requirement 5: Server Actions — Orders

**User Story:** As a developer, I want `createOrder` and `getOrderByNumber` Server Actions, so that checkout can place orders atomically and confirmation pages can retrieve them.

#### Acceptance Criteria

1. THE `lib/actions/orders.ts` module SHALL export a `createOrder` function that accepts a `CreateOrderInput` and returns an `Order`.
2. WHEN `createOrder` is called, THE `orders` action SHALL execute all database writes (Order creation, OrderItem creation, stock decrements, coupon increment) within a single database transaction.
3. WHEN `createOrder` is called and any variant has `stock` less than the requested `quantity`, THE `orders` action SHALL roll back the entire transaction and throw an `InsufficientStockError`.
4. WHEN `createOrder` is called with a valid, active, non-expired coupon code within its usage limit, THE `orders` action SHALL apply the discount and increment the coupon's `usedCount` by 1.
5. WHEN `createOrder` is called with a subtotal exceeding the free-shipping threshold, THE `orders` action SHALL set `shippingCost` to `0`.
6. WHEN `createOrder` is called with a subtotal at or below the free-shipping threshold, THE `orders` action SHALL set `shippingCost` to the flat shipping rate.
7. THE `lib/actions/orders.ts` module SHALL export a `getOrderByNumber` function that accepts an `orderNumber` string and returns an `Order` or `null`.
8. WHEN `createOrder` receives invalid input, THE `orders` action SHALL reject the call with a Zod validation error before performing any database operation.

---

### Requirement 6: Server Actions — Admin

**User Story:** As an admin, I want Server Actions for managing products, orders, brands, and categories, so that I can perform CRUD operations from the admin dashboard.

#### Acceptance Criteria

1. THE `lib/actions/admin.ts` module SHALL export `adminGetProducts`, `adminCreateProduct`, `adminUpdateProduct`, `adminDeleteProduct`, and `adminToggleProductVisibility` functions.
2. THE `lib/actions/admin.ts` module SHALL export `adminGetOrders` and `adminUpdateOrderStatus` functions.
3. THE `lib/actions/admin.ts` module SHALL export `adminGetBrands`, `adminCreateBrand`, `adminUpdateBrand`, and `adminDeleteBrand` functions.
4. THE `lib/actions/admin.ts` module SHALL export `adminGetCategories`, `adminCreateCategory`, `adminUpdateCategory`, and `adminDeleteCategory` functions.
5. WHEN any admin mutating action is called, THE `admin` action SHALL validate input with Zod before performing any database operation.
6. WHEN `adminToggleProductVisibility` is called, THE `admin` action SHALL set `isVisible` to the logical inverse of its current value and return the updated Product.
7. WHEN `adminUpdateOrderStatus` is called with a valid `OrderStatus` value, THE `admin` action SHALL update the order's `status` field and return the updated Order.

### Requirement 7: Customer Schema and Order Model Update

**User Story:** As a developer, I want a `Customer` model in the Prisma schema and an updated `Order` model, so that orders are linked to customers rather than authenticated users.

#### Acceptance Criteria

1. THE `prisma/schema.prisma` file SHALL define a `Customer` model with fields: `id`, `firstName`, `lastName`, `email` (unique), `phone` (unique), `address`, `city`, `state`, `zip`, `createdAt`, `updatedAt`, and a `orders` relation.
2. THE `prisma/schema.prisma` file SHALL define an `OrderStatus` enum with values `PENDING`, `PROCESSING`, `SHIPPED`, `DELIVERED`, and `CANCELLED`.
3. THE `Order` model SHALL replace the `userId`/`User` relation with a `customerId`/`Customer` relation.
4. THE `Order` model SHALL use the `OrderStatus` enum for its `status` field with a default of `PENDING`.
5. THE `Order` model SHALL include a `shippingCost` field of type `Float` with a default of `0`.

---

### Requirement 8: Checkout Integration

**User Story:** As a customer, I want to complete a two-step checkout flow, so that I can provide my details and place an order that is confirmed with an order number.

#### Acceptance Criteria

1. WHEN a customer submits checkout step 1, THE checkout flow SHALL call `upsertCustomer` with the provided name, email, phone, and address fields.
2. WHEN `upsertCustomer` succeeds, THE checkout flow SHALL store the returned `customerId` and advance to step 2.
3. WHEN a customer confirms the order in step 2, THE checkout flow SHALL call `createOrder` with the stored `customerId`, cart items, and any applied coupon code.
4. WHEN `createOrder` succeeds, THE checkout flow SHALL redirect the user to `/orders/confirmation?order={orderNumber}`.
5. IF `createOrder` throws an `InsufficientStockError`, THEN THE checkout flow SHALL display a toast notification indicating the affected item is no longer available in the requested quantity.
6. WHEN checkout step 1 is rendered, THE checkout form SHALL use Zod schema validation via `react-hook-form` with `@hookform/resolvers`.

---

### Requirement 9: Search Overlay and URL Routing

**User Story:** As a customer, I want to search for products using the header overlay and see results on a dedicated page, so that I can find products by name, brand, or category.

#### Acceptance Criteria

1. WHEN a user submits a query in the `SearchOverlay`, THE `SearchOverlay` SHALL navigate to `/search/[encoded-query]` using `router.push`.
2. THE `app/search/[query]/page.tsx` file SHALL be a React Server Component that calls `searchProducts(decodeURIComponent(params.query))`.
3. WHEN the Search_Page renders with results, THE Search_Page SHALL display a grid of matching products.
4. WHEN the Search_Page renders, THE Search_Page SHALL display a `ProductRecommendation` section below the results grid.
5. WHEN `searchProducts` returns an empty `results` array, THE Search_Page SHALL display a "no products found" message for the query.
6. WHEN `searchProducts` returns an empty `results` array, THE Search_Page SHALL still render the `ProductRecommendation` section with available recommendations.

---

### Requirement 10: Responsive Cart Sheet

**User Story:** As a customer, I want the cart to appear as a bottom drawer on mobile and a side sheet on desktop, so that the cart is easy to use on any device.

#### Acceptance Criteria

1. WHEN the cart is opened on a viewport narrower than 640px, THE `CartSheet` SHALL render the cart contents inside a `Drawer` component (vaul) that slides up from the bottom.
2. WHEN the cart is opened on a viewport 640px or wider, THE `CartSheet` SHALL render the cart contents inside a `Sheet` component with `side="right"`.
3. THE `CartSheet` SHALL use the `useIsMobile` hook from `hooks/use-mobile.ts` to determine the current breakpoint.
4. WHEN the viewport width changes across the 640px breakpoint, THE `CartSheet` SHALL switch between `Drawer` and `Sheet` without requiring a page reload.

---

### Requirement 11: Cookie-Based Wishlist

**User Story:** As a customer, I want to save products to a wishlist stored on my device, so that I can revisit and act on saved items without needing an account.

#### Acceptance Criteria

1. THE `lib/wishlist-context.tsx` module SHALL provide a `WishlistContext` with `items`, `addItem`, `removeItem`, `toggleItem`, `isInWishlist`, and `clearWishlist`.
2. WHEN `addItem` is called with a `WishlistItem`, THE `WishlistContext` SHALL add the item to the wishlist and write the updated list to the `wishlist` cookie.
3. WHEN `removeItem` is called with a `variantId`, THE `WishlistContext` SHALL remove the matching item from the wishlist and write the updated list to the `wishlist` cookie.
4. WHEN `toggleItem` is called with a `WishlistItem`, THE `WishlistContext` SHALL add the item if it is not present or remove it if it is already present.
5. THE wishlist cookie SHALL have a `max-age` of 30 days, `path=/`, and `SameSite=Lax`.
6. FOR ALL valid `WishlistItem[]` arrays, writing the array to the cookie and then reading it back SHALL produce an array deeply equal to the original.
7. WHEN the `wishlist` cookie is absent or contains malformed JSON, THE `WishlistContext` SHALL initialise `items` to an empty array.
8. THE `components/wishlist-sheet.tsx` component SHALL render a checklist of wishlist items with a checkbox per item.
9. WHEN "Remove selected" is activated in the `WishlistSheet`, THE `WishlistSheet` SHALL call `removeItem` for every checked item.
10. WHEN "Add selected to cart" is activated in the `WishlistSheet`, THE `WishlistSheet` SHALL add every checked item to the cart context and then remove those items from the wishlist.
